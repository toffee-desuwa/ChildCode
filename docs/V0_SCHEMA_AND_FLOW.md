# ChildCode — v0 最小 Block Schema 与页面流程

## A. v0 Block Inventory

v0 只允许 4 类块，全部为白名单，不开放自由文本输入。

### subject（对象）
猫、狗、机器人、小女孩、小男孩、龙、兔子、宇航员

### action（动作）
奔跑、飞翔、画画、跳舞、睡觉、吃东西、弹吉他

### scene（场景）
森林、太空、海底、城堡、沙漠、雪山、糖果王国

### style（风格）
水彩、像素风、油画、卡通、蜡笔画、剪纸风

**约束：**
- v0 不做数量块（如"3 只猫"）
- v0 不做重复块、条件块、逻辑块
- 每类块每次只选 1 个值

## B. Block Data Schema

### 单个块的最小字段

```json
{
  "type": "subject | action | scene | style",
  "value": "白名单中的一个值",
  "label": "展示给孩子的中文名"
}
```

### Blockly → JSON 真相层的最小结构

```json
{
  "version": "v0",
  "blocks": {
    "subject": { "value": "cat", "label": "猫" },
    "action": { "value": "running", "label": "奔跑" },
    "scene": { "value": "forest", "label": "森林" },
    "style": { "value": "watercolor", "label": "水彩" }
  }
}
```

### 示例 1 — 第一次生成

```json
{
  "version": "v0",
  "blocks": {
    "subject": { "value": "cat", "label": "猫" },
    "action": { "value": "running", "label": "奔跑" },
    "scene": { "value": "forest", "label": "森林" },
    "style": { "value": "watercolor", "label": "水彩" }
  }
}
```

### 示例 2 — 只改一个块后的第二次生成

将 scene 从"森林"改为"太空"：

```json
{
  "version": "v0",
  "blocks": {
    "subject": { "value": "cat", "label": "猫" },
    "action": { "value": "running", "label": "奔跑" },
    "scene": { "value": "space", "label": "太空" },
    "style": { "value": "watercolor", "label": "水彩" }
  }
}
```

## C. Prompt Derivation Rule

系统从 JSON 真相层派生底层 prompt 的基本规则：

1. 读取 JSON 中的 `subject`、`action`、`scene`、`style` 四类核心信息
2. 按固定模板组合成一句自然语言描述，作为图片生成 API 的输入
3. MVP 默认只使用这四类核心信息参与派生，不引入额外修饰或参数
4. 孩子不可见派生后的 prompt 文本

派生过程对孩子完全透明——孩子只看到积木和生成的图片，不接触中间的 prompt。

## D. Page Flow

### Step 1 — 进入应用
- **用户看见：** 欢迎页 / 入口页
- **能做什么：** 选择"开始创作"或进入家长配置
- **系统反馈：** 若 API key 未配置，引导先完成家长配置

### Step 2 — 家长最小配置
- **用户看见：** 家长最小配置入口（页面或弹窗），包含 API key 输入与额度设置
- **能做什么：** 输入 API key、设定使用额度
- **系统反馈：** 验证 API key 有效性，保存配置，提示配置完成
- **备注：** 此步骤只需首次完成，之后孩子可直接进入搭建区

### Step 3 — 孩子进入搭建区
- **用户看见：** Blockly 工作区，左侧展示四类积木分类（对象、动作、场景、风格）
- **能做什么：** 从每类中拖拽 1 个积木块到工作区
- **系统反馈：** 实时显示已选积木；若四类未选齐，"生成"按钮不可用

### Step 4 — 生成第一张图
- **用户看见：** 四类积木已选齐，"生成"按钮可用
- **能做什么：** 点击"生成"按钮
- **系统反馈：** 显示加载状态 → 展示生成的图片（图 A）

### Step 5 — 引导只改一个块
- **用户看见：** 图 A 展示中，工作区积木仍保留，出现提示"试试只改一个块，看看会有什么不同？"
- **能做什么：** 替换工作区中的任意一个积木块
- **系统反馈：** 高亮被替换的块，"再次生成"按钮出现

### Step 6 — 生成第二张图
- **用户看见：** 修改后的积木组合 + "再次生成"按钮
- **能做什么：** 点击"再次生成"
- **系统反馈：** 显示加载状态 → 展示第二张图片（图 B）

### Step 7 — 对比两张图
- **用户看见：** 图 A 和图 B 并排展示，被替换的积木块高亮标注
- **能做什么：** 观察差异；可继续修改积木开始新一轮对比
- **系统反馈：** 标注哪个块被替换了，帮助孩子建立"改了什么 → 变了什么"的因果联系

## E. Edge States

### 缺少必填块
- 四类积木未全部选齐时，"生成"按钮禁用
- 提示文案："还差 X 类积木哦，把它们都拖过来吧！"

### API key 未配置
- 孩子尝试进入搭建区时，跳转到家长配置页
- 提示文案："请让爸爸妈妈先完成设置"

### 额度耗尽
- "生成"按钮禁用
- 提示文案："创作次数已用完，请让爸爸妈妈查看设置"

### 生成失败
- 展示友好的错误提示："图片生成遇到了一点问题，再试一次？"
- 提供"重试"按钮
- 不向孩子暴露技术错误细节

### 第二次生成时改了不止一个块
- 系统不强制阻止（避免挫败感）
- 但在对比视图中明确标注所有被修改的块
- 提示文案："你改了 N 个块哦。试试只改 1 个，更容易看出区别！"

## F. Scope Freeze Reminder

v0 明确不做：

- 不做自由文本输入
- 不做数量、重复、条件、逻辑块
- 不做积木嵌套或组合逻辑
- 不做课程系统 / 教学引导流程
- 不做 harness / 测试管理 UI
- 不做商业化 / 付费 / 账户体系
- 不做多语言
- 不做社交 / 分享功能
- 不做 prompt 术语教学
- 不做复杂家长管理 UI（只做最小配置）
- 不做图片编辑 / 保存 / 导出
- 不做用户注册 / 登录
